## ⚡ Phase 1. 사용자 경험(UX) 및 온보딩 개선 (완료)

가장 시급하고 구현 난이도가 낮은 항목들을 중심으로 개선을 완료했습니다.

### [1-A] 온보딩 가이드 및 접근성 강화 (Problem 1-1)
- **배경**: 첫 실행 시 API 키 발급 경로를 알지 못해 초기 진입 장벽이 높았음.
- **설정 바로가기**: Gemini 및 Jira 설정 카드 상단에 API 키/토큰 발급 페이지로 즉시 이동할 수 있는 링크를 추가했습니다.
- **네트워크 가이드**: 사내망에서 접속이 원활하지 않은 Gemini 발급 사이트에 대해 "모바일/테더링 권장" 안내 문구를 명시했습니다.
- **배치 개선**: 링크 위치를 카드 최상단으로 옮겨 시인성을 높였습니다.

### [1-B] Jira 연결 상태 확인 기능 (Problem 1-3)
- **배경**: 입력한 토큰의 유효성을 확인할 방법이 없어 연동 실패 시 원인 파악이 어려웠음.
- **즉시 검증**: 입력한 PAT Token이 유효한지 서버를 통해 즉시 확인하는 '⚡ 연결 확인' 버튼을 구현했습니다.
- **상태 표시**: 연결 성공 시 사용자의 이름(DisplayName)과 확인 시간을 표시하여 신뢰도를 높였습니다.

### [2-A] Gemini 대화 히스토리 영속화 (Problem 2-1)
- **배경**: 페이지 새로고침 시 대화 내용이 모두 사라져 연속적인 작업이 불가능했음.
- **브라우저 저장**: 채팅 내역을 `localStorage`에 자동 저장합니다.
- **유지력**: 브라우저를 새로고침하거나 서버를 재시작해도 이전 대화 내용을 그대로 이어갈 수 있습니다.
- **관리 기능**: 필요 시 대화 내역 전체를 삭제할 수 있는 '대화 초기화' 기능을 제공합니다.

### [3-C] 이슈 인라인 액션 (Problem 1-2, 3-2, 3-5, 4-7)
- **배경**: 이슈를 확인한 뒤 수정하거나 Jira 원본으로 이동하기 위해 탭을 전환해야 하는 번거로움이 있었음.
- **댓글 추가**: 유사 이슈 검색 결과 카드에서 `💬 댓글` 버튼 클릭 시 인라인 텍스트 입력창이 등장하고, 바로 Jira에 댓글을 등록할 수 있습니다. (`submitComment()` → `/api/jira-update`)
- **상태 변경**: `🔄 상태` 버튼 클릭 시 해당 이슈의 사용 가능한 Transition 목록을 드롭다운으로 표시하며, 선택 후 즉시 상태를 변경합니다. (`toggleTransitionForm()` → `/api/jira-transitions` → `submitTransition()`)
- **Jira 외부 링크**: `🔗 Jira` 버튼으로 `jira.sinc.co.kr/browse/{key}`에 직접 접속 가능합니다.

### [5-C] 친절한 에러 메시지 가이드 (Problem 1-4, 5-5)
- **배경**: 시스템 에러 발생 시 단순 코드만 노출되어 사용자가 후속 조치를 취하기 어려웠음.
- **상황별 힌트**: Jira API 호출 실패 시 단순 코드(401, 403)만 보여주는 대신, "토큰 만료", "권한 부족" 등 사용자 레벨에서 조치할 수 있는 안내를 병렬로 표기합니다.

## 🚀 Phase 2. 핵심 기능 추가 (진행 중)

### [4-A] 캐시 구축 진행률 표시 (SSE) (Problem 4-2) ✅
- **배경**: 수백 건의 임베딩 작업 중 진행 상태를 알 수 없어 오류나 멈춤으로 오인하는 경우가 발생함.
- **실시간 피드백**: 임베딩 작업 중 어떤 이슈를 처리하고 있는지, 전체 대비 몇 %가 진행되었는지를 SSE(Server-Sent Events)를 통해 실시간으로 보여줍니다.
- **부하 방지**: 시스템 성능 유지를 위해 1회 구축 시 **최대 100건**까지만 처리하도록 제한을 추가했습니다.
- **안정성**: 작업 중 오류가 발생해도 중단되지 않고 다음 이슈로 넘어가며, 최종 결과를 요약하여 리포트합니다.

### [3-A] 자연어 → JQL 검색 통합 (Problem 3-1, 5-6) ✅
- **배경**: JQL 문법 숙련도에 따라 검색 기능 활용도 격차가 매우 컸음.
- **Agent Query**: Gemini 채팅 탭에서 자연어로 "이번달 완료한 변경관리 이슈 보여줘"와 같이 입력하면 `api_gemini_process_agent()`가 의도(SEARCH)를 분석하고 JQL을 자동 생성합니다.
- **자동 실행**: 생성된 JQL이 Jira 테스트 탭으로 자동 전달되어 검색이 실행됩니다.
- **투명한 동작**: 생성된 JQL과 분석 Reason을 Gemini 채팅에 표시하여 사용자가 동작 과정을 확인할 수 있습니다.

### [2-C] 맥락 주입형 Jira 채팅 (Problem 2-3) ✅
- **배경**: 채팅과 Jira 조회가 분리되어 있어 이슈 정보를 일일이 복사해서 질문해야 했음.
- **Agent 통합**: Gemini 채팅 탭에서 이슈 관련 요청 시 Agent Query가 자동으로 의도(ACTION)를 분석하여 이슈 키·상태 변경·댓글 내용을 파라미터로 추출합니다.
- **승인 기반 실행**: 이슈 수정 액션은 카드 형태로 미리보기를 제공하고, 사용자가 '승인 및 실행' 버튼을 클릭해야만 실제로 Jira에 반영됩니다. (`appendAgentActionCard()` → `executeAgentAction()`)
- **안전 장치**: 의도 분석과 실행을 분리하여 오작동 위험을 최소화했습니다.

### [4-C] 유사 이슈 → 처리 초안 자동 생성 (Problem 1-2, 4-7) ✅
- **배경**: 유사 이슈를 찾아도 내용을 직접 훑어보고 댓글을 수동으로 작성해야 하는 휴먼 노력이 여전히 컸음.
- **AI 초안 생성**: AI검증 완료 후 `✍ 처리 초안 작성` 버튼을 클릭하면, Gemini가 현재 미해결 이슈와 최적 매칭된 완료 이슈의 상세 내용을 분석하여 댓글 초안을 자동 생성합니다.
- **편집 가능**: 생성된 초안은 편집 가능한 텍스트 영역에 표시되어 사용자가 필요에 따라 수정할 수 있습니다.
- **즉시 등록**: `📝 Jira에 댓글 등록` 버튼으로 초안을 그대로 Jira에 댓글로 달 수 있어 창 전환 없이 작업이 완료됩니다.
- **구현**: `api_draft_comment()` (gemini.py) + `/api/draft-comment` 엔드포인트 + `draftComment()`/`submitDraft()` (script.js)

### [UI-E] 유사 이슈 검색 결과 유형별 그룹화 ✅
- **배경**: 검색 결과에 서로 다른 이슈 유형이 섞여 있어 업무 우선순위 판단이 어려웠음.
- **개선 내용**:
  - **그룹화**: `issuetype` 필드를 기준으로 결과를 자동 분류하여 섹션별로 출력합니다.
  - **시각적 헤더**: 각 섹션 상단에 유형 이름과 건수를 표시하여 현황을 한눈에 파악하게 했습니다.
  - **아코디언 토글**: 섹션 헤더 클릭 시 내용을 접거나 펼칠 수 있어 화면 공간을 효율적으로 사용합니다.
- **구현**: `searchSimilar()`, `buildTypeSection()`, `toggleTypeSection()` (script.js) + `.type-section` 스타일 (style.css)

## 🛠 인프라 및 배포 환경 개선 (완료)

팀 단위 배포 및 원활한 운영을 위한 기반 작업을 완료했습니다.

### [4-B 인프라] 관리자용 Docker 배포 가이드 개편
- **문서화**: 기존 가이드를 관리자 관점에서 서비스 준비, 보안 설정, 생명주기 관리로 세분화하여 개편했습니다. (`04_관리자용_Docker배포가이드.md`)

### [4-C 인프라] 컨테이너 기반 배포 자동화
- **Docker 지원**: `Dockerfile` 및 `docker-compose.yml`을 작성하여 명령어 하나로 전체 환경을 구축할 수 있게 했습니다.
- **환경 변수 관리**: `.env` 파일을 통해 민감한 키 값을 안전하게 분리 관리하는 체계를 구축했습니다.

### [5-D 인프라] 멀티유저 및 보안 강화 (Problem 5-4)
- **배경**: 단일 서버 설정으로 인해 팀원들과 공유 시 개인별 API 키 사용이 불가능했음.
- **개인 키 분리**: API 키와 토큰을 브라우저 `localStorage`에 저장하여 각 사용자가 독립적으로 자신의 키를 관리합니다.
- **서버 무상태**: 서버가 특정 사용자의 키를 보관하지 않으므로 팀 공용 배포에 적합합니다.
- **하위 호환**: 기존 `.env`/`claude.json` 방식과 병행하여 로컬 개발에도 문제 없습니다.
